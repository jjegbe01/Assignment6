NodeJS

A Javascript library that does more.

Posted by Julian Egbert on February 23, 2015

NodeJS Introduction

Last post, we covered the basics of AngularJS. Today, we'll be looking at some of the basics of another important tool: NodeJS.

Node.js has a formal definition as follows:

"Node.js is an open source, cross-platform runtime environment for server-side and networking applications. Node.js applications are written in JavaScript, and can be run within the Node.js runtime on OS X, Microsoft Windows, Linux, FreeBSD, and IBM i."

More specifically, it's an event-driven framework based on and built ontop of Chome's built-in javascript engine. However, its functionality extends far beyond that of a traditional javascript library. Not only does it have all the functionality brought about by javascript, but it also features a good amount of code specifically allowing it to interface with most commonly found servers or filesystems.

However, there's even more that sets it apart. Node.js is often labeled as a non-blocking I/O system. This means that it can handle multiple activities concurrently without ever making a program wait/sleep. Ontop of that, it can also handle these concurrent activities without resorting to using up multiple threads. This really sets it apart from more traditional languages, which are based heavily on the idea of sequential execution.

To demonstrate this clearly, we'll use a code sample pulled from a wonderful Node.js tutorial hosted over at airpair.

Here's a chunk of sample PHP we would use to populate a webpage.

$response = file_get_contents("http://example.com");
print_r($response);

Below is a chunk of code that achieves the same outcome, but using Node.js instead.

var http = require('http');

http.request({ hostname: 'example.com' }, function(res) {
  res.setEncoding('utf8');
  res.on('data', function(chunk) {
    console.log(chunk);
  });
}).end();

Since the Node.js example handles data in chunks, as opposed to sleeping until it recieves a response after making a request, it allows the system to handle other activities concurrently. This makes it ideal for real time applications which are reliant on being able to respond to frequent user input.
